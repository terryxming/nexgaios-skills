# Audit Rubric

用本文件执行 audit mode。目标不是把 skill 说得更漂亮，而是发现它在真实 Codex 使用中为什么会失效。

## 优先级

- **P0 阻塞**：会导致 skill 不触发、严重误触发、发布物不可追溯、用户被误导、执行危险操作或输出不可用。
- **P1 重要**：会导致体验不稳、维护困难、上下文浪费、测试缺失、输出质量不可预测。
- **P2 优化**：表达、结构、示例、报告格式、复盘材料可以更清晰，但不阻塞当前使用。

每条问题必须包含：优先级、视角、证据、后果、修复建议。没有证据就降级为观察或假设。

## 资深 AI 产品经理视角

检查这个 skill 作为 AI 产品能力是否成立：

- 目标用户是否明确：维护者、运营人员、设计师、开发者还是普通用户。
- 任务是否具体：是否聚焦某类高频、复杂、易错或需要专门知识的工作。
- 触发是否清楚：frontmatter `description` 是否覆盖用户自然语言、关键对象和边界。
- 成功标准是否可判断：输出物、质量门槛、验收方式是否清晰。
- 能力边界是否合理：是否过宽、过窄、和已有 skill 重叠，或把普通提示包装成 skill。
- 用户价值是否成立：是否减少重复劳动、降低错误、稳定输出或沉淀组织知识。
- 失败路径是否清楚：缺少输入、权限不足、外部系统不可用、数据不可信时如何处理。

常见问题：

- 只描述“做得更好”，没有定义好到什么程度。
- 把内部实现流程当成用户价值。
- `description` 太泛，导致不会触发或到处误触发。
- 没有说明何时不该使用这个 skill。

## 生产端视角

检查这个 skill 是否便于构建、维护、验证和迭代：

- `SKILL.md` 是否短、准、可执行，是否把细节拆到 references。
- frontmatter 是否只有 `name` 和 `description`，且 `description` 承担触发责任。
- references 是否按主题拆分，是否存在重复、过期或未路由资料。
- scripts 是否真的解决确定性问题，是否有运行说明和基础验证。
- assets 是否只放输出所需资产，不塞说明文档或临时产物。
- 流程自由度是否合适：高风险步骤是否有门禁，低风险步骤是否避免过度约束。
- 是否有验证命令、测试样例、forward-test 策略和失败处理。
- 是否有环境假设：路径、系统、账号、网络、第三方工具是否写清。
- 是否有安全边界：生产系统、凭据、隐私数据、 destructive 操作是否受控。

常见问题：

- `SKILL.md` 变成百科全书，加载后挤占上下文。
- reference 文件存在但主入口没有说明何时读取。
- 脚本没有测试，或者脚本行为和 skill 指令不一致。
- 修改 skill 时没有同步 `skill.yaml`、`CHANGELOG.md` 或 `agents/openai.yaml`。

## 消费端视角

检查真实用户使用这个 skill 时是否顺畅：

- 用户用自然语言请求时是否会触发，例如“帮我挑刺这个 skill”“检查能不能发版”。
- 第一次使用是否不需要理解内部术语。
- 缺少输入时，是否只问关键问题，而不是一次问太多。
- 输出是否按严重程度排序，并能直接指导下一步。
- 报告是否区分确定缺陷、风险、建议和主观偏好。
- 是否给出具体文件、行号、片段或可复现 prompt。
- 语气是否像帮用户做质量把关，而不是泛泛批评。
- 是否能处理草案、完整目录、单文件、两个版本对比等常见输入形态。

常见问题：

- 输出只给原则，没有可执行修复。
- 报告太长但没有 Top Issues。
- 只站在作者视角，忽略真实用户会怎么说、怎么误用。
- 未说明哪些风险未验证。

## 必测 prompt 类型

- 理想触发：`使用 $skill-doctor 审计 skills/foo/bar`
- 模糊触发：`帮我看看这个 skill 有没有问题`
- 发布触发：`检查这个 skill 能不能发布 v0.2.0`
- 对比触发：`对比 v0.1.0 和 v0.2.0 的变化`
- 误触发边界：`帮我写一个普通 README`，不应强行进入 skill 审计。
