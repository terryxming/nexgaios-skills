# Lifecycle Policy

用本文件执行 release mode，或在 audit mode 中检查版本治理。

## 核心原则

已发布版本必须冻结。任何行为、触发、流程、reference、script、asset、输出契约或测试基准的变化，都必须通过新版本承载。

不要把“当前文件历史”当成版本治理。一个 released 版本应当是可追溯、可回滚、可测试、可复盘的发布物。

## 生命周期状态

- `draft`：允许频繁修改，不承诺稳定。
- `candidate`：发布候选，只允许修阻塞问题、补测试、补记录。
- `released`：已发布，必须冻结，不允许原地修改。
- `deprecated`：仍可追溯和回滚，但不推荐新任务使用。
- `retired`：历史保留，仅用于审计、复盘或回滚参考。

如果目标仓库只支持 `active` 等较粗状态，也要在诊断报告中补充推断的发布状态，并说明证据。

## 版本号规则

使用 SemVer：`MAJOR.MINOR.PATCH`。

- `MAJOR`：触发条件、核心工作流、输出契约、目录结构或用户迁移成本发生破坏性变化。
- `MINOR`：新增审计维度、报告模块、测试策略或脚本能力，兼容旧用法。
- `PATCH`：修正文案、补充边界说明、修复脚本小 bug，不改变行为契约。

这些变化都必须进入版本记录：

- `SKILL.md` frontmatter `description` 变化。
- `SKILL.md` 正文流程、门禁或输出要求变化。
- `references/` 内容变化。
- `scripts/` 行为变化。
- `assets/` 模板或输出资产变化。
- `agents/openai.yaml` 面向用户的名称、描述、默认 prompt 变化。
- 输出格式、评分标准、测试基准、发布门禁变化。

## CHANGELOG 规则

`CHANGELOG.md` 必须存在，从 `v0.1.0` 开始记录。时间线倒叙，最新版本放在最前面。

推荐格式：

```markdown
# 更新日志

## v0.2.0 - 2026-06-17

### 新增
- ...

### 变更
- ...

### 修复
- ...

### 移除
- ...

### 测试
- ...

### 生命周期
- 状态：released。
- 冻结：是。
- 回滚目标：v0.1.0。
```

每个版本至少覆盖：新增、变更、修复、移除、测试、生命周期。无内容时写“无”，不要删除栏目。

## 发布门禁

发布前逐项检查：

- `skill.yaml` 版本号与目标发布版本一致。
- `release.tag` 与 `<skill-id>@<version>` 一致。
- `CHANGELOG.md` 最新条目对应目标版本，且位于最前。
- `SKILL.md` frontmatter 可触发、无占位文本。
- `agents/openai.yaml` 若存在，和当前定位一致。
- 结构验证通过，例如 `pnpm skill:validate <skill-id>`。
- 至少完成触发测试、误触发测试、真实 prompt 测试、输出契约测试。
- 如果是非 patch 版本，完成至少一次 forward-test 或说明为什么跳过。
- 记录回滚目标和兼容性风险。

## 回滚规则

回滚不是编辑旧版本，而是恢复到某个已冻结发布物或重新发布一个修复版本。

报告中必须说明：

- 可以回滚到哪个版本。
- 回滚会丢失哪些能力或修复。
- 是否需要同步安装目录或重新发布。
- 是否存在数据、脚本、外部依赖不兼容。

## 复盘规则

每次发布后都应能回答：

- 为什么改。
- 改了什么。
- 影响谁。
- 怎么测。
- 怎么回滚。
- 哪些风险留到后续版本。
