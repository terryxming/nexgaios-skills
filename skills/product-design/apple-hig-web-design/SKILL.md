# Apple HIG Web Design

用这个 skill 把 Apple Human Interface Guidelines 转译成可执行的前端视觉改造。核心目标不是“仿 Apple”，而是在保留产品任务的前提下，提升清晰度、层级、材质、组件状态、响应式和可访问性。

## 渐进式披露

只加载当前任务真正需要的材料：

1. 先读本文件，明确流程和引用路由。
2. 根据页面类型与修改范围，选择性读取 `references/` 中的文件。
3. 只有在要复用 token、组件片段或运行检查时，才使用 `assets/` 与 `scripts/`。

不要一次性读取所有引用；除非用户明确要求更新 skill、审计完整规范，或任务横跨 BDD、SDD、视觉系统、组件和测试验证。

## 执行流程

1. 识别页面目标、主要用户任务、产品类型、技术栈、视口范围和当前视觉债。默认视口范围是桌面+移动；如果用户明确说“电脑端”“桌面端为主”“无需移动端”，则把移动端排除在本轮验收之外。
2. 先诊断，不改文件：像专业 UI/UX 设计师一样，从信息架构、视觉层级、布局、组件、颜色、字体、间距、可访问性、已纳入范围的视口、动效和内容密度拆到原子级问题。
3. 按需读取引用：
   - `references/bdd-scenarios.md`：需要把用户诉求转成验收行为时读取。
   - `references/sdd-architecture.md`：需要做系统化改造、多页面改造或架构拆分时读取。
   - `references/design-principles.md`：做大范围视觉层级、交互取舍或 UX 判断前读取。
   - `references/visual-system.md`：改 token、字体、颜色、间距、圆角、阴影、材质或暗色模式前读取。
   - `references/liquid-glass.md`：添加模糊、透明、半透明导航、悬浮控制或玻璃材质前读取。
   - `references/components.md`：改导航、侧边栏、标签栏、搜索、按钮、表单、卡片、列表、工具栏、弹窗或空状态前读取。
   - `references/testing-evals.md`：验证、截图、checkpoint、eval 或最终交付前读取。
   - `references/source-index.md`：用户要求“最新/当前 Apple 指南”、材料时效重要，或更新本 skill 时读取。
4. 输出带版本号的改造方案，例如 `Apple HIG Visual Redesign Plan v1.0`。方案必须包含问题诊断、原子级问题清单、改造目标、适用视口范围、分模块策略、验收标准、风险与不改动边界。
5. 等待用户明确确认具体方案版本。只有用户说“确认 v1.0”“按 v1.0 执行”“可以开始改”等明确授权后，才进入代码改造。
6. 如果用户提出疑问、修改或反馈，先解决反馈，再生成新版本方案。小修订递增 `v1.1`，范围或方向变化递增 `v2.0`；继续等待确认。
7. 只基于用户确认的方案版本实现。优先使用项目已有组件和 token；只有项目缺失设计系统时，才新增局部 token 层。
8. 运行页面，在已确认的视口范围内检查，并迭代到文本、控件、焦点、材质层级和布局稳定性都成立。只有当移动端在范围内时，才把移动端横向溢出或移动导航作为阻塞项。

## 方案确认门禁

- 未获得用户对具体版本方案的明确确认前，禁止创建、修改、覆盖或格式化目标页面/项目文件。
- 允许为了诊断读取文件、运行只读检查、打开页面和截图；如果某项诊断会写文件，必须先说明并征得同意。
- 用户确认的是某个版本的方案，不是泛泛授权。执行时不得偷偷扩大范围。
- 用户反馈优先级高于旧方案。收到反馈后必须先回答或吸收反馈，再输出新版本方案并重新请求确认。
- 最终执行总结要说明“基于已确认的哪个方案版本”完成。

## 默认契约

- 保留业务行为和用户原有工作流，除非用户明确要求改变 UX 流程。
- Apple-like 是设计质量和平台直觉，不是 Apple 品牌化；不要使用 Apple logo、产品截图、专有资产或暗示从属关系。
- 运营工具、SaaS、CRM、数据报告和仪表盘要保持安静、密集、可扫读，不要套营销页大 hero。
- Liquid Glass 只用于表达层级、导航或前景控制，不要把整页或所有卡片都做成玻璃。
- 所有改动都要考虑键盘焦点、文本溢出、暗色模式、动效降级和已纳入范围视口的布局稳定性。触控尺寸与移动端横向溢出只在移动端进入范围时作为验收项；如果用户明确排除移动端，只记录为范围外风险。

## BDD / SDD / TDD 快速约束

- BDD：先把“视觉变好”改写成可验证行为，例如主任务更明显、导航更稳定、状态更清楚、文本不溢出。
- SDD：把改造拆成目的层、结构层、token 层、组件层、交互层、验证层，避免散乱堆样式。
- TDD/tests：只在交互行为或组件契约变化时补测试；纯视觉改造优先跑现有 lint/build/test 和浏览器 QA。
- checkpoints：页面目的、验收行为、视口范围、token、组件状态、布局稳定性、可访问性、最终验证要逐项过关。
- evals：按 `references/testing-evals.md` 的评分表检查，有阻塞项先修再交付。

## 实现资源

- `assets/apple-web-tokens.css`：项目没有设计 token 时，可作为 Apple HIG 风格 token 参考；适配到项目命名后再使用。
- `assets/component-recipes.md`：常见 CSS/HTML/React 组件片段，用前先适配项目结构和类名。
- `scripts/apple_hig_static_check.py <project-root>`：视觉改造后的建议性静态检查；它不能替代浏览器截图和人工判断。

## 反模式

- 整页模糊、全局玻璃、卡片套卡片、所有区块都浮起来。
- 单一蓝紫、深蓝灰、米棕、咖啡橙等一眼偏色的配色。
- 在任务型页面使用巨型营销 hero。
- 文字依赖视口单位缩放，或在按钮/卡片里裁切、重叠、难以换行。
- 动效阻塞任务、隐藏状态变化，或忽略 `prefers-reduced-motion`。
- 用 Apple 品牌、产品名或专有资产做装饰。
