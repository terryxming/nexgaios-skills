# 组件

修改常见 UI 组件前读取本文件。

## 导航与 App Shell

- 顶层目的地要稳定可见。
- 侧边栏适合承载集合、文件夹、项目或应用区域。
- 标签栏适合同级顶层区块，尤其是移动端。
- 通过选中态、标题、面包屑或 active tab 明确当前位置。
- 桌面端不要轻易把主导航藏进菜单，除非空间确实不足。

## 侧边栏

- 使用安静材质或色调背景。
- 图标尺寸和标签对齐保持一致。
- 侧边栏图标要语义化；除非颜色编码类别，否则避免彩色噪声。
- 内容可在视觉上延伸到侧边栏下方，但必须保证可读性和滚动稳定。
- 只有在标签仍可发现时，才提供折叠行为。

## 标签栏

- 用于少量顶层区域。
- 标签保持简短。
- 选中态必须清楚。
- 移动端标签触控目标要舒适稳定。
- 不要把无关动作混进标签导航。

## 搜索

- 放在用户预期的位置：工具栏、页头、侧边栏，或以搜索为核心产品的专用 tab。
- Placeholder 可放示例，不要替代必要 label。
- 支持清除、加载、空结果和无结果状态。
- 搜索范围重要时，使用 token 或筛选器表达。
- 结果尽量靠近搜索框。

## 按钮

- 每个区域只保留一个主动作。
- 只有常见动作或已有 tooltip/label 时，才使用纯图标按钮。
- 破坏性动作要视觉区分，高风险时要确认。
- 补齐 hover、active、focus-visible、disabled 和 loading 状态。
- 按钮文字短且稳定。

## 表单

- 使用可见 label。
- 帮助文本靠近字段。
- 适合时做 inline validation，但不要过早制造噪声。
- 出错后保留用户已输入内容。
- 必填/可选状态不能只靠颜色表达。

## 卡片与面板

- 卡片用于重复项、预览和有边界的对象。
- 不要把卡片当作通用页面分区容器。
- 除非内层是真正重复对象，否则避免卡片套卡片。
- 圆角适中，内容层级清楚。

## 表格与列表

- 数字和日期对齐一致。
- 筛选和排序状态保持可见。
- 谨慎使用 row hover 和 selected 状态。
- 密集数据区域不要重模糊或重阴影。
- 提供空、加载和错误状态。

## 工具栏

- 相关动作分组。
- 熟悉图标配 label 或 tooltip。
- 主要命令靠近受影响内容。
- 紧凑工具栏里，标准图标优先于长文本按钮。

## Dialog、Sheet 与 Popover

- Modal dialog 用于阻塞进度的决策。
- Sheet 或 panel 用于聚焦编辑或检查。
- 提供清楚的关闭/取消行为。
- 破坏性确认要明确。
- 只在 modal 打开时 trap focus，并在关闭后恢复焦点。

## 反馈状态

- Loading：进度显示在受影响区域附近。
- Success：尽量不打断地确认，除非结果需要注意。
- Error：说明发生了什么以及如何恢复。
- Empty：说明这里应该出现什么，并提供下一步有用动作。
