# SDD 架构

当视觉改造不只是改几个颜色，而是需要形成可维护的小系统时读取本文件。

## 目标

建立一个可复用的 Apple HIG-informed 前端改造过程：把产品目标转成代码层改动，同时保护可用性、可访问性和可维护性。

## 输入

- 用户请求和期望结果。
- 现有页面或应用代码。
- 前端框架与样式栈。
- 产品类型：运营工具、移动 app shell、落地页、内容站、电商、游戏或原型。
- 已提供的品牌约束。
- 支持设备、断点和本轮视口范围；如果用户明确桌面优先或排除移动端，要把移动端标为范围外。
- 现有测试、构建和开发命令。

## 输出

- 执行前的带版本号改造方案。
- 在现有技术栈内完成的局部代码改动。
- 视觉 token 的新增或调整。
- 组件状态、响应式和交互细节改进。
- 已确认视口范围内的浏览器验证记录。
- 可用时的静态检查结果。
- 最终总结：改了什么、跑了什么验证、剩余风险是什么。

## 分层设计

1. 诊断层：拆解信息架构、视觉层级、布局、组件、色彩、字体、间距、可访问性、已纳入范围的视口、动效和内容密度问题。
2. 目的层：定义主任务、内容层级和应降噪的元素。
3. 结构层：选择导航、布局网格、响应式行为和安全内容区域。
4. Token 层：定义字体、颜色、间距、圆角、阴影、材质和暗色模式。
5. 组件层：细化按钮、搜索、标签、侧边栏、表单、卡片、列表、弹窗和反馈状态。
6. 交互层：补齐状态、过渡、动效规则、焦点和输入 affordance。
7. 验证层：运行测试、检查截图、打 eval 分，并修复阻塞项。

## 方案状态机

1. `Draft v1.0`：基于页面诊断生成初版方案。
2. `Review`：等待用户确认、提问或反馈。
3. `Revised v1.x/v2.0`：吸收反馈后生成新版本方案，并回到 `Review`。
4. `Confirmed vx.x`：用户明确确认具体版本。
5. `Executing`：只按已确认版本改造代码。
6. `Verified`：完成测试、截图和最终说明。

未进入 `Confirmed` 状态前，不能修改目标项目文件。

## 接口契约

这个 skill 不绑定具体框架，应适配：

- CSS Modules、Sass、普通 CSS、Tailwind、shadcn/ui、Radix、MUI、Chakra、自定义 React/Vue/Svelte 组件或静态 HTML。
- 项目已有设计 token。
- 仅浅色或浅/深色双模式实现。

## 约束

- 除非用户明确要求 UX 流程变化，否则保留现有业务行为。
- 优先使用本地组件和 token，避免为了视觉改造新增依赖。
- 不把 Apple 商标、logo、产品图当通用装饰使用。
- 避免整页玻璃、失控模糊、低可读透明度和过重阴影。
- 已纳入范围的视口都要保持文本可读、控件稳定。用户明确桌面-only 时，移动端问题只作为范围外风险记录。
- 可行时尊重 reduced motion 和 reduced transparency fallback。

## 检查点

1. 发现：页面目的、技术栈和主要 UI 风险已理解。
2. 范围：桌面、移动、暗色模式等本轮视口/模式范围已明确。
3. 诊断：原子级 UI/UX 问题已拆解，且优先级只针对本轮范围。
4. 方案：带版本号的改造方案已输出。
5. 确认：用户已明确确认具体版本。
6. 规格：BDD 验收行为已写出或明确在心智模型里。
7. Token：字体、颜色、间距、圆角和表面层级一致。
8. 组件：被改组件的状态、语义和可访问性完整。
9. 布局：已纳入范围的视口布局稳定。
10. 可访问性：焦点、标签、对比度和 reduced motion 已检查。
11. 最终：测试、浏览器截图和 eval 评分完成。

## 风险处理

- 页面已有品牌时，把 Apple-like 清晰度融入品牌，而不是替换品牌。
- 数据密集型应用优先处理对齐、层级、表格可读性和筛选，而不是大图和氛围。
- 已有 CSS 框架时，按该框架习惯实现。
- 用户明确桌面-only 时，优先优化桌面密度、侧边栏、表格、证据阅读和键鼠交互；不要把移动适配作为本轮阻塞项。
- 无法启动服务时，检查静态文件并说明验证边界。
- 如果 Apple 当前指南会影响结论，先读取 `source-index.md` 并核对官方来源。
