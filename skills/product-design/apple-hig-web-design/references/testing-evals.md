# 测试、检查点与评估

最终验证前读取本文件。它定义 TDD、自动化测试、浏览器 QA 和评分标准。

## TDD 循环

1. 从 BDD 场景定义验收检查。
2. 确认已有用户明确批准的方案版本。
3. 确认本轮视口范围；用户明确桌面-only 时，不把移动端纳入阻塞验收。
4. 找出现有测试命令：package scripts、lint、typecheck、build、unit、integration、Storybook、visual tests。
5. 只有交互行为改变或组件契约需要保护时才新增测试。
6. 实现视觉变化。
7. 运行自动化检查。
8. 用浏览器截图做 QA。
9. 按 eval rubric 打分并修复阻塞项。

## 执行前门禁

开始改文件前必须同时满足：

- 已输出带版本号方案。
- 用户已明确确认该版本。
- 当前执行范围没有超出该版本方案。
- 视口范围已写入方案；桌面-only、移动端、暗色模式等范围没有被擅自扩大。
- 如用户提出过反馈，已生成新版本并重新确认。

## 自动化测试

运行项目已有命令：

- `npm test`、`pnpm test`、`yarn test` 或框架对应单元测试。
- `npm run lint` 或等价命令。
- TypeScript 项目运行 `npm run typecheck`。
- 可用时运行 `npm run build`。
- UI 流程变化时运行已有 Playwright/Cypress 测试。

不要为了狭窄的纯样式任务引入重型测试栈。只有交互行为变化时，才补聚焦测试。

## 建议性静态检查

运行：

```bash
python <skill-dir>/scripts/apple_hig_static_check.py <project-root>
```

把提示视为需要检查的线索。脚本能捕捉常见问题，但不能替代浏览器 QA。

## 浏览器 QA 检查点

- 桌面视口：如果页面主要用于电脑端，至少检查一个常用宽屏视口；重点看层级、导航、主动作、间距、表格和文本换行。
- 移动视口：只有移动端在本轮范围内时才检查 tab、侧边栏折叠、触控目标、表单、sticky bar 和横向溢出。
- 暗色模式：只有项目已有暗色模式、用户要求暗色模式，或方案明确纳入暗色模式时才作为阻塞验收。
- 键盘：tab 穿过被改控件，确认 `focus-visible`。
- Reduced motion：确认动效被关闭或简化。
- 长内容：测试长标签、表格行、错误文本和空状态。
- 真实数据：不要只用理想 placeholder 判断视觉效果。

## 评估量表

每项 0-2 分：

- 流程合规：先诊断和出方案，确认具体版本后才执行。
- 目的清晰：主任务能立刻被理解。
- 层级：布局能引导注意力且不杂乱。
- 熟悉感：控件符合可识别的 Web/平台模式。
- 视觉工艺：对齐、间距、字体、状态和表面质量有意图。
- 材质克制：模糊/透明有用途且可读，不是装饰滥用。
- 组件完整：被改组件有状态和语义。
- 视口稳定：已纳入范围的视口稳定；被用户明确排除的视口不计为扣分项。
- 可访问性：label、focus、对比、动效和触控目标可接受。
- 可维护性：改动贴合现有技术栈，避免无意义依赖。

通过标准：

- 不能有任何 0 分项。
- 大型改造总分至少 17/20。
- 小范围改动中，受影响项应尽量拿 2 分，除非用户明确要快速草稿。

## 阻塞问题

最终回复前必须修复：

- 主动作或导航更难发现。
- 未经用户确认具体方案版本就修改目标文件。
- 文本重叠、裁切或不可读。
- 玻璃效果降低可用对比度。
- 已纳入范围的视口出现非预期横向滚动。
- 被改交互控件没有可见键盘焦点。
- 现有 build 或核心测试因改造失败。

## 最终回复清单

说明：

- 视觉与结构改了什么。
- 基于已确认的哪个方案版本执行。
- 本轮验证覆盖了哪些视口；哪些视口被用户明确排除。
- 触及的文件。
- 跑过哪些测试/检查。
- 做过哪些浏览器/截图验证，或为什么无法做。
- 剩余风险或后续建议。
