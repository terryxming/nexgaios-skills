# 设计原则

当需要判断“改什么、保留什么、删什么”时读取本文件。内容基于 Apple HIG 设计原则主题转译为 Web UI 工作规则。

## 目的

- 首次扫视时，让主目标显而易见。
- 优先呈现用户真正来完成的功能，弱化次要动作。
- 移除与任务竞争的纯装饰 UI。
- 通过内容、层级和工艺表达产品个性，而不是靠噱头。

## 主体性

- 让用户无需被多余流程牵引，就能抵达内容和动作。
- 逃离路径要清楚：返回、关闭、取消、撤销、重置或编辑。
- 避免模态陷阱和隐藏导航。
- 当用户成果有风险时，提供可逆操作和恢复路径。

## 责任

- 请求数据或权限时说明清楚理由。
- 避免暗黑模式、意外订阅或含糊的破坏性动作。
- 清楚表达加载、错误、数据使用和 AI 输出不确定性。
- 保护用户创造的内容，避免意外丢失。

## 熟悉感

- 优先使用用户熟悉的 Web 和平台约定，再考虑自定义控件。
- 重复元素在页面间保持一致。
- 状态必须可见：选中、当前位置、禁用、加载、已更改、无效。
- 图标要符合常见含义；不熟悉的图标搭配标签或 tooltip。

## 灵活性

- 面向多种输入方式设计：鼠标、触控、键盘、读屏。
- 响应式布局不能隐藏关键任务。
- 控件在不同断点保持可预测位置。
- 用自然过渡帮助用户理解布局变化。

## 简洁

- 简洁是聚焦有用，不是空洞极简。
- 标签要短，重复控件要少。
- 相关动作和内容放在一起。
- 优先直接操作和清楚 affordance，而不是靠说明文字补救。

## 工艺

- 边缘、基线、图标尺寸和间距要有意图地对齐。
- 打磨状态、微交互、空/错/加载状态。
- 动画顺滑但克制。
- 用真实内容长度测试，不只看理想 demo 文案。

## 愉悦

- 个性要强化产品目的。
- 材质、颜色、动效和节奏感要少量使用。
- 愉悦不能阻塞任务完成。
- 质感来自稳定细节：焦点、反馈、层级和恢复能力。

## Web 转译

- 追求真实 UI 改善，而不是视觉模仿。
- 让内容贴近用户，让控制随手可得。
- 用深度解释交互层：基础内容、导航、上下文控制、modal/sheet。
- 色彩保持克制，使强调和状态仍有意义。
