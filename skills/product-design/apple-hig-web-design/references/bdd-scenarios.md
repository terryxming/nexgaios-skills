# BDD 场景

当用户提出“Apple 风格”“HIG 质感”“高级视觉改造”这类需求时，先把抽象审美目标转成可验收行为，再开始改代码。

## 功能：Apple HIG 驱动的前端视觉改造

```gherkin
功能: Apple HIG-informed web UI redesign
  作为 产品负责人或前端构建者
  我想要 页面以 Apple HIG-informed clarity and polish 被改造
  以便 体验更有用、熟悉、响应式且精致
```

## 场景：保留产品目的

```gherkin
假如 一个已有真实用户工作流的前端页面
当 页面被改造成 Apple-like UI/UX
那么 用户的主任务必须更容易被看见和开始
而且 原有业务行为必须保持 intact
而且 装饰性变化不能遮挡控件或内容
```

验收关注：

- 页面第一眼能判断“这里要做什么”。
- 主操作、导航、筛选、状态和关键数据没有被弱化。
- 视觉增强不改变数据、交互和跳转语义。

## 场景：先诊断并等待确认

```gherkin
假如 用户要求改造一个页面
当 skill 触发后
那么 Codex 必须先阅读和观察页面
而且 从信息架构、视觉层级、布局、组件、色彩、字体、间距、可访问性、响应式、动效和内容密度拆解原子级问题
而且 输出带版本号的改造方案
而且 在用户明确确认该版本前不能修改目标文件
```

验收关注：

- 初版方案使用 `v1.0`。
- 用户提出小反馈后输出 `v1.1`、`v1.2` 等修订版。
- 用户改变方向、范围或目标后输出 `v2.0` 等主版本。
- 执行只能基于用户明确确认的方案版本。

## 场景：桌面优先或移动端排除

```gherkin
假如 用户明确说明页面主要在电脑端使用，或无需处理移动端
当 Codex 诊断和制定方案
那么 问题优先级、改造策略和验收标准必须以桌面视口为准
而且 移动端问题不得作为阻塞项
而且 方案中必须写明移动端属于范围外或非本轮目标
```

验收关注：

- 桌面端至少检查一个常用宽屏视口，例如 1440px。
- 移动端可作为已知风险记录，但不进入 P0/P1 任务清单。
- 不为了移动端重构桌面信息架构，除非用户后来把移动端重新纳入范围。

## 场景：运营工具或仪表盘

```gherkin
假如 页面是 SaaS、dashboard、CRM、admin、analytics 或 productivity surface
当 应用 Apple HIG polish
那么 结果应当安静、密集、可扫读
而且 导航、筛选、表格和状态指示仍然突出
而且 设计应避免巨型营销 hero 和装饰性卡片堆叠
```

验收关注：

- 信息密度和可扫读性优先于氛围。
- 表格、指标、筛选和动作区稳定可见。
- 卡片只用于真正的对象、摘要或重复项。

## 场景：落地页或品牌页

```gherkin
假如 页面是 public landing、portfolio、venue 或 product page
当 应用 Apple HIG polish
那么 首屏应清楚展示品牌、产品、人物或具体 offer
而且 辅助文案应解释价值主张
而且 下一屏内容应在折叠线附近有可见暗示
```

验收关注：

- H1 是品牌、产品、人名、地点或明确 offer，不把价值描述塞成空泛标题。
- 首屏需要真实或生成的相关视觉资产，而不是纯渐变或抽象 SVG。
- 已纳入范围的视口都能看到下一段内容的线索。

## 场景：移动 Web App

```gherkin
假如 页面必须在移动端工作，或用户没有排除移动端
当 导航、标签、搜索或表单被重设计
那么 触控目标应舒适
而且 标签不能裁切或重叠
而且 顶层导航应可预测
而且 键盘焦点和读屏语义应保持有效
```

验收关注：

- 移动端在范围内时，390px 宽度下无意外横向滚动。
- 粘性导航、底部栏或工具栏不遮挡主要内容。
- 长标签、错误文本和真实数据不会撑破布局。

## 场景：Liquid Glass 请求

```gherkin
假如 用户要求 Liquid Glass 或半透明 Apple-style surfaces
当 添加材质效果
那么 玻璃应限制在前景控件、导航、工具栏、sheet 或上下文面板
而且 底层内容不能降低可读性
而且 设计在 reduced transparency 或不支持 backdrop-filter 时仍然可用
```

验收关注：

- 玻璃是层级和交互信号，不是全页滤镜。
- 文本对比度、边框、遮罩和 fallback 都成立。
- 暗色模式下不靠模糊承担所有分层。

## 场景：可访问性与动效

```gherkin
假如 一个被改造后的页面
当 执行 QA
那么 它应支持键盘导航
而且 有可见 focus states
而且 有足够对比度
而且 尊重 reduced motion preferences
而且 文本缩放后布局不破裂
```

验收关注：

- 修改过的交互控件都有 `:focus-visible`。
- 动效是辅助理解，不阻塞任务。
- 状态不只靠颜色表达。

## 场景：最终验证

```gherkin
假如 代码修改完成
当 最终验证运行
那么 可用时应通过 build 或 lint
而且 页面应在已确认的视口范围中完成视觉检查
而且 交付前应修复 blocking eval issues
```

验收关注：

- 说明测试、截图、静态检查和限制。
- 如果无法运行某项验证，要明确说明原因。
