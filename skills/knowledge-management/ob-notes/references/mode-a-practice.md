# Mode A · 实战型笔记

适用于"一次性解决的具体问题"：某个报错怎么修、某命令/配置怎么用、某个坑怎么绕过。本质是**一条带时间戳的事件记录**，解决了就基本冻结，价值在于**具体到三个月后能照着直接复用**。

格式细节一律遵循 `frontmatter-tags.md`，本文件只规定实战型特有的结构与取舍。

## 何时用实战型（vs 研究型）

- **实战型**：问题具体、解完不再变、不需要长期生长。例：Shift+Tab 在某版本跳过 plan mode 怎么办、某 npm 包安装报错怎么修。→ 一事一笔记、带日期、写完冻结。
- **研究型**：主题会反复回来深挖。→ 走 `mode-a-research.md`。

判据：这事解决了之后我还会回来"补充扩展"吗？基本不会、只会偶尔"查阅复用" → 实战型。

## 该记什么（实战型的灵魂是"细节足够复用"）

铁律二在这里尤其关键——**确切信息一字不改地留下**：

- **问题现象**：确切的报错原文、触发条件、环境（OS/版本/工具）。
- **根因**：为什么会这样（哪怕只是当时的判断，标可信度）。
- **解法**：确切的命令、配置、步骤，原样保留，能直接复制执行。
- **没走通的弯路**（可选）：试过但无效的方案，省得未来重蹈。

最容易翻车的是把解法概括掉。反例：把"`claude --upgrade` 后重启会话即恢复"压成"升级一下就好"——后者无法复用。

## 该丢什么

- 解决过程里的来回猜测、情绪、客套。
- 与该问题无关的发散。
- 能在解法里一句话覆盖的背景。

## 命名与冻结

- 按 naming-rule：主题 + 日期，如 `claude-plan-mode-2026-06-25.md`，写完冻结。
- 实战型通常一次成形、不追加。若同一问题日后有新解法/新情况，优先**新建一条带新日期的笔记**，并与旧的双链关联，而非改写旧记录——保留"当时确实这么解的"事实。
- 例外：若只是订正旧笔记里的明确错误，按 datestamp-rule 留痕修正。

## 与研究型的边界（交叉时）

实战中若沉淀出了**可推广的通用知识**（不止解决这一次），那部分按交叉场景处理（见 SKILL.md mode-decision）：实战记录为主落点，通用知识另起一篇研究型笔记，两者双链回指。
