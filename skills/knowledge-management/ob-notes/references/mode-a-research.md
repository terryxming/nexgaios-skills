# Mode A · 研究型笔记

适用于"一个主题一路深挖"的知识沉淀：问某个概念是什么、沿着提问/思考/查证不断展开、并且这个主题日后还会回来继续补充。本质是**一篇会持续生长的主题知识库**，不是一次性记录。

格式细节（frontmatter 字段、中文 tag、可信度标记、日期/过时标注、双链/callout、命名）一律遵循 `frontmatter-tags.md`，本文件不重复定义，只规定研究型特有的结构与取舍。

## 何时用研究型（vs 实战型）

- **研究型**：主题会反复回来、需要长期追加、值得织进知识网。例：深挖 loop engineering 是什么、RL 最新进展、某框架的设计哲学。→ 一主题一笔记，长期追加。
- **实战型**：一次性解决的具体问题，解完基本不再变。例：某报错怎么修、某命令怎么用。→ 走 `mode-a-practice.md`。

判不准时，问一句：这个主题三个月后我还会回来补充吗？会 → 研究型；不会 → 实战型。

## 该记什么（命中才留）

逐条扫对话，命中以下信号才沉淀：

- 概念的**本质定义**与关键区分（它是什么、不是什么）。
- 推理链与**为什么**：得出某结论的思路，不只是结论。
- 关键事实、数据、引用来源（保留确切出处，呼应铁律二）。
- 不同观点/方案的**对比与取舍**。
- 你自己的判断、疑问、待验证的点（这些是后续生长的钩子）。

## 该丢什么

- 寒暄、过程性的来回试错。
- 最终被推翻的中间结论（除非演化过程本身有价值，那就按 datestamp-rule 留痕）。
- 能一句话带过的常识。
- agent 的客套与重复表述。

## 笔记骨架

研究型笔记套这个可生长的骨架。各块按需出现，但 30 秒读法和正文是必备。**文件名即标题，正文不写 H1**（见 frontmatter-tags 的 naming-rule、frontmatter-spec）：

```markdown
---
title: {{主题名，如 Loop Engineering}}
date: {{创建日 YYYY-MM-DD}}
updated: {{最后更新日}}
source: {{来源 agent/任务}}
source_url: {{原文/一手出处链接，有外部来源就填，见 frontmatter-spec}}
tags: [类型/研究, 可信/待验证, 状态/持续]
read_count: 0
last_read:
---

> [!summary] 30 秒读法
> - **是什么**：{{这篇讲的东西}}
> - **解决什么**：{{针对什么问题}}
> - **最重要结论**：{{单条最该带走的判断}} `[可信度]`
> - **怎么用**：{{怎么落地复用}}
> - **前置**：{{要看懂这篇得先掌握 / 学会什么；没有真前置就省}}
> - **边界**：{{能做什么、做不了什么（能力速览，非可信度）；适用范围的细化见正文「适用边界」}}

## 是什么
{{本质定义、关键区分。每条结论标可信度，如 `[已验证]`}}

## 为什么 / 怎么来的
{{推理链、背景、动机}}

## 关键点
- {{要点，确切事实/数据保留原样}} `[已验证]`
- {{要点}} `[待验证]`

## 适用边界
{{这套结论在什么范围内成立、什么情况下不适用——别让未来的你误用}}

## 下次怎么用
{{下次遇到什么情况会回来用这篇？给判断准则或可迁移模式，不只是回忆历史}}

## 我的疑问 / 待验证
- {{留给后续生长的钩子}}

## 关联
- [[{{相关主题}}]]  {{建链前按 linking-convention 查目标是否存在，未建标 (待建)}}
```

## source-fidelity：源材料保真（材料是网页/长文/报告时必做）

当沉淀对象是一篇外部材料（文章、博客、白皮书、长报告、视频）时，最大的风险是**把长文压成一张观点卡**——只留"我的启发"，丢了原文的问题定义、分类、方法步骤、例子、框架对比维度和限制条件。那样三个月后这篇笔记既不能当知识、也不能当资料回查。两道动作钉死它：

**① 正文开头放来源元数据块**（比单个 source_url 更完整，给人直接可读的出处）：

```markdown
> [!info] 来源
> **原文链接**：
> **原文标题**：
> **作者**：
> **原文发布日期**：
> **笔记日期**：
```

字段能识别就如实填，识别不到写"不适用"，**不要猜**（猜出来的作者/日期是伪信息，违反铁律三）。

**② 给一张原文结构覆盖表**：**默认保留类的章节枚举到最细层级（有 H3 就到 H3）；整节省略的（营销/履历）可 H2 一行带过**。把 N 个子节折成一行 = 省略了 N-1 节——每行标"压缩/省略"的，**"原因"必须点名丢掉的具体内容**（如"未收 Mem0 六层模块名"）；写"拆成表""理论压掉"**不算理由**。目的仍是把"有意取舍"和"无意丢失"分开。

```markdown
## 原文结构覆盖

| 原文部分 | 处理方式 | 原因 |
|---|---|---|
| {{问题定义 / 前言}} | 保留 | 是文章的问题定义，不能只留结论 |
| {{框架细节}} | 压缩 | 留比较维度，细节拆成要点 |
| {{作者履历}} | 省略 | 对本次复用价值低 |
```

判据：默认**保留**原文的问题定义、分类体系、方法步骤、**机制怎么运作**、关键例子、方案对比维度、限制/边界、可复用资料链接（含**确切的模块名 / 工具名 / 参数**，呼应铁律二）；默认**可压缩**营销介绍、作者履历、重复解释、与目标无关的细节。

**放行用实质闸，不用字数比率**：逐个被标"压缩"的技术 / 框架 / 方法，自问——**它的核心机制，能不能只凭这篇笔记读懂、不回原文？** 有一个读不懂 = 压过头，补回机制层。**覆盖表是记账手段，不是免压通行证。**（落盘不足原文主干 1/4 时，几乎必然有机制被压没，实质闸要逐项严过。）

**源上限护栏**：只补原文**确实讲了**的；原文没展开的深层（算法 / 实现 / 协作流程）**不许编**（铁律三），改写进"我的疑问 / 待验证"当待深挖钩子，日后另开研究扩展、从一手文档源码补。

## 生长与防腐化

研究型笔记会反复追加，重点防止它腐化成自相矛盾的旧信息：

- **追加带日期**：新内容按 datestamp-rule 标日期、刷新 `updated`。
- **推翻旧结论时留痕不删**：用删除线 + 过时标注（见 datestamp-rule），保留认知演化。
- **状态轴维护**：还在深挖标 `状态/持续`；告一段落标 `状态/稳定`；发现可能过期标 `状态/待复核`。
- **双链织网**：相关主题之间建 `[[双链]]`（linking-convention），让研究型笔记连成网而非孤岛。新建笔记时主动找已有相关主题回链。

## 增量 vs 新建

沉淀前先在知识库找同主题笔记（按 naming-rule 的主题命名便于查找）：

- 找到同主题笔记 → **追加**到对应小节，按生长规则带日期。
- 没有 → 新建。避免就同一主题攒出多篇碎片。

## 掌握视角：让笔记驱动掌握，而非只记录（mastery-lens）

研究型笔记的目标不是"记下我读过什么"，而是"让未来的我真正掌握这个主题"。掌握不是"我知道这个概念"，而是我能**解释它、判断它、应用它、迁移它**。落笔前用这几个问题自检——答得出，就把答案**织进正文该在的位置**；答不出，说明还没吃透，先想清楚再写：

- **能讲清吗**：三五句话能讲给一个外行听懂吗？（讲不清＝没真懂）
- **何时不成立**：它的边界、前提、偏见在哪？
- **能迁到哪**：除了眼前这个场景，这套思路还能用在别处吗？
- **连得上谁**：它和我已有的哪篇笔记相关？

> [!warning] 这是写作的内功，不是笔记的骨架
> 上面四问是**作者的思考纪律，绝不能变成笔记里的固定章节**（如"## 复述""## 迁移"）。把方法论摆成标题，会让人为填空而填空、写出表演式的伪掌握，也把整合好的理解又切碎了。掌握只能从**字里行间的功力**透出来：开篇立场写得像"你会怎么教别人"、迁移作为一句自然的旁注出现、关联是真实的双链——而不是贴标签。

适用范围：本纪律只服务**研究型 / 持续生长**的知识笔记。一次性的实战笔记（mode-a-practice）不必套用，否则是过度设计。
