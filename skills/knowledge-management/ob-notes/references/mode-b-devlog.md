# Mode B · 项目开发日志（dev-log）

适用于**项目记忆**：某个项目"当时为什么这么做、做到哪了、踩了什么坑"。它和知识沉淀（Mode A）本质不同——复用者是"未来继续做这个项目的你和 agent"，目的是**让一个隔了很久才回来的项目能无缝续上、不推翻自己、不重踩旧坑**。

落点由 `preflight.md` 的 landing-rule 决定（跟所属项目走），本文件不重复。格式细节遵循 `frontmatter-tags.md`。

## 核心机制：哪些块追加、哪些块覆盖

dev-log 是单文件、随项目生长。关键在于区分两类块（呼应 datestamp-rule）：

- **只追加、不删**：进展时间线、决策表、踩坑记录——这些是历史，保留演化痕迹。
- **覆盖更新**：当前状态 / 下一步——它表达"此刻进展"，每次收工直接刷新，不堆叠历史。

这个区分让你既有"最新状态可直接续"，又有"完整演化史可回溯"。

## 续做协议

每次回到这个项目时：
1. **先读 dev-log**（铁律五），恢复上下文：看"当前状态/下一步"知道续哪、看"决策表"知道哪些不能推翻、看"踩坑"知道哪些别再踩。
2. 干活。
3. **收工前更新**：刷新"当前状态/下一步"，追加这次的新决策、新踩坑、给进展时间线加一条。

## 该记什么

- **意图**：这个项目要解决什么、成功长什么样（近乎不变）。
- **决策 + 理由 + 否决的备选**：尤其"为什么要做这次迭代"这类最容易忘、忘了最可怕的。
- **进展**：每次开工做了什么、为什么。
- **踩坑**：现象/根因/解法，保留确切命令报错（铁律二）。
- **当前卡点与下一步**：让未来的自己直接接上。

## 该丢什么

- 已经写进代码、看代码就懂的实现细节（dev-log 记"为什么"，不复述"是什么"）。
- 过程性闲聊。

## dev-log 骨架

```markdown
---
title: {{项目名}} 开发日志
date: {{创建日}}
updated: {{最后更新日}}
source: {{来源 agent}}
tags: [类型/项目日志, 状态/持续]
read_count: 0
last_read:
---

# {{项目名}} 开发日志

## 项目意图
{{要解决什么、成功标准。近乎不变}}

## 架构与关键决策（只追加）
| 日期 | 决策 | 理由 | 否决的备选 |
|---|---|---|---|
| {{YYYY-MM-DD}} | {{做了什么决定}} | {{为什么}} | {{考虑过但放弃的，及为何}} |

## 当前状态 / 下一步（覆盖更新）
- 现状：{{做到哪了}}
- 下一步：{{接下来要做什么}}
- 卡点：{{当前卡在哪，若有}}

## 进展时间线（只追加，倒序）
- {{YYYY-MM-DD}}：{{这次做了什么、为什么}}

## 踩坑记录（只追加）
- {{YYYY-MM-DD}} {{现象}} — 根因：{{原因}} `[已验证]` — 解法：{{确切命令/配置}}
```

## devlog-integration：与 CLAUDE.md / AGENTS.md 打通

dev-log 记"历史与为什么"，CLAUDE.md / AGENTS.md 记"现状与规则"（技术栈、约定、怎么跑），两者职责不同、不可混。打通方式很轻——**在项目的 CLAUDE.md / AGENTS.md 里加一行指向 dev-log**：

```markdown
继续本项目前，先读 docs/dev-log.md（或本项目 dev-log 落点）了解决策背景与当前进度。
```

这样 agent 每次开工读配置文件时会被引导去读 dev-log，复用了"agent 必读 CLAUDE.md"这个现成机制，又不把 dev-log 的体积压进配置文件。**不要把 dev-log 内容塞进 CLAUDE.md**——配置文件要小而稳，dev-log 会持续生长。

## 与 CHANGELOG 的边界（若项目要发布）

dev-log 面向维护者，详尽记"为什么、怎么踩坑"；CHANGELOG 面向用户，精简记"版本间变了什么"。两者职责不同、不算重复。本 skill 自身的维护也遵循此分工（见 maintenance.md 版本规则）。
