# 反例库 — 坏笔记与改写

本文件只给**坏例 → 为什么坏 → 怎么改**，用于抽取与过滤噪音时对照自查。它**不重新定义**可信度、双链、日期等规则——命中处一律按各自唯一家处理（如可信度按 `frontmatter-tags.md` 的 credibility-spec）。

这些坏例正是本 skill 要钉死的"四类毛病"（不知道记什么 / 压缩过狠 / 格式丑 / 没重点）的具体长相。命中任一，先重写结构，再谈版式。

## 1. 流水账（没重点）

> 坏：先做了 A，报错；又试 B、C；最后发现可能是 D。

为什么坏：根因、模式、下次动作全被过程噪音淹没；未来检索命中后还得重读一遍。

改写：主体改成 **症状 → 根因 → 解法 → 防复发**；过程压成一行时间线或"排除过的路径"。

## 2. 空泛总结（压缩过狠的另一面）

> 坏：这次说明流程很重要，以后要加强测试、注意沟通。

为什么坏：正确但不可执行——没有触发条件、证据、边界、下一步。

改写：写清"**什么时候**触发这条经验"，把口号改成检查清单。

## 3. 过度保留（压缩不足）

> 坏：粘贴完整聊天 / 完整命令输出 / 完整工具日志，每个尝试同等权重。

为什么坏：信息密度低，阅读成本高，高价值判断没被编辑出来。

改写：原始证据只留**能支撑判断**的片段，其余写成"无效尝试摘要"。注意这与铁律二不冲突——铁律二要留的是确切的命令/报错/版本号等**可复用细节**，不是聊天过程。

## 4. 过度提炼（压缩过狠）

> 坏：系统设计要保持单一事实源，并建立良好的反馈闭环。

为什么坏：抽象正确，但丢了原始场景——未来不知道这条适用于什么问题。这是模型最爱犯的"本能压缩"。

改写：保留**触发场景 + 反例 + 适用边界**；原则后必须跟一个具体"下次怎么用"。长文/网页材料尤其要防压成观点卡——见 `mode-a-research.md` 的 source-fidelity。

## 5. 格式炫技（格式丑的另一面）

> 坏：callout、表格、标签、双链很多，但同一结论在摘要/时间线/索引里重复，读者仍找不到主线。

为什么坏：版式没有服务结构，装饰反而打断阅读。

改写：每个版式组件只承担一个职责——表格用于比较与状态，callout 只用于摘要/风险/边界。具体约定见 linking-convention。

## 6. Agent-only（把人挡在门外）

> 坏：全篇都是"下个 agent 应读取 X、执行 Y、检查 Z"，用户看不到故事线与判断依据。

为什么坏：笔记只服务机器续作，人不愿读，也就不会回访。

改写：**第一屏写给人看**（是什么 / 解决什么 / 最重要结论）；给 agent 的操作指令降级到"下次怎么用 / 恢复入口"。判断依据要留，不能只留指令。

## 一条贯穿原则

未经验证的内容别写成确定结论——按 credibility-spec 标 `[待验证]` 或放进"适用边界 / 待验证"，不要为了笔记"显得完整"而把推测固化成事实。
