# 评测与保真度

本文件用于评估 `obsidian-knowledge-curator` 的业务能力。结构验证不等于业务能力正确。

## 评测模式

| 模式 | 目标 | 通过标准 |
| --- | --- | --- |
| 真实 prompt 测试 | 验证自然语言触发和输出质量。 | 能按用户意图产出冻结预览，不静默写入。 |
| 误触发测试 | 验证普通任务不会被强行沉淀。 | 没有授权时只建议，不写文件。 |
| 输出契约测试 | 验证标题、冻结编号、元数据、结构和安全边界。 | 所有必填结构存在，且没有未确认字段。 |
| 保真度测试 | 验证网页/长文没有压缩过狠。 | 原文主章节有覆盖表，关键信息未静默丢失。 |
| 版式测试 | 验证 Obsidian 可读性。 | 版式服务结构，不靠装饰制造复杂感。 |
| 视觉渲染测试 | 验证 CSS snippet、callout、Mermaid 和表格是否真的可见可读。 | 按 `obsidian-visual-patterns.md` 说明验证级别，不能只检查文件内容。 |
| forward-test | 验证其他 agent 能否按 skill 稳定执行。 | 新 agent 在未见过样例上仍能遵守确认闸门和保真度规则。 |

## 网页/长文保真流程

对网页、博客、白皮书、长报告执行沉淀时：

1. 抽取原文标题、作者、发布日期和主要 H2/H3 结构。
2. 生成原文结构覆盖表。
3. 判断每个章节的处理方式：保留、压缩、拆分、省略。
4. 省略任何主章节时说明理由。
5. 再写 30 秒读法和二次提炼。
6. 对照原文和预览，检查是否只留下观点而丢失材料。

## 原文结构覆盖表

```markdown
## 原文结构覆盖

| 原文部分 | 处理方式 | 原因 |
| --- | --- | --- |
| 前言 / 当前大语言模型的困境 | 保留 | 是文章问题定义，不能只保留结论。 |
| 主流框架细节 | 压缩 | 保留比较维度，细节可拆分为工具卡。 |
| 作者介绍 | 省略 | 对本次知识复用价值低。 |
```

## 信息保真判断

沉淀不是全文转载，但也不是只保留启发。

默认保留：

- 原文的问题定义。
- 原文的分类体系。
- 原文的方法步骤。
- 原文的重要例子。
- 原文的框架/方案对比维度。
- 原文的限制、边界和适用场景。
- 对用户有直接复用价值的资料链接。

默认可以压缩：

- 营销性介绍。
- 作者履历。
- 重复解释。
- 与用户目标无关的服务细节。
- 不影响理解的代码长片段，但要保留其用途和关键信号。

默认不要静默删除：

- 原文主章节。
- 业务场景例子。
- 方法步骤。
- 框架的关键机制。
- 与用户当前设计有关的限制条件。

## 失败样例

原文有完整结构：

```text
LLM 困境 -> 记忆类型 -> 记忆产生/策略/存储/检索 -> 上下文工程 -> 框架分析 -> 资料链接
```

坏沉淀只剩：

```text
记忆不是存储，而是上下文调度
```

问题：这是一张观点卡，不是网页 raw。它丢失了原文作为资料的证据层和复用层。
