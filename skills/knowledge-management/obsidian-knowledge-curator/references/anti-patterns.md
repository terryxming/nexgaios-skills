# 反例库

反例用于约束输出边界。命中任一反例时，先重写结构，再考虑版式。

## 流水账反例

坏例特征：

```markdown
先做了 A，然后报错。后来试了 B，又试了 C。最后发现可能是 D。
```

为什么坏：

- 用户看不到根因、模式和下次动作。
- 有效信息被过程噪音淹没。
- 未来检索命中后仍要重新读一遍全部过程。

改写策略：

- 把过程压缩成时间线或证据。
- 主体改成“症状 -> 根因 -> 解法 -> 防复发”。

好例片段：

```markdown
> [!summary] 30 秒读法
> 这次问题不是命令本身失败，而是路径所在 drive 不存在。下次遇到 Windows 路径异常，先用 `Test-Path` 和 drive 检查排除环境问题。
```

## 空泛总结反例

坏例特征：

```markdown
这次说明流程很重要，以后要加强测试，注意沟通。
```

为什么坏：

- 正确但不可执行。
- 没有触发条件、证据、边界和下一步。

改写策略：

- 写清“什么时候触发这个经验”。
- 把口号改成检查清单。

好例片段：

```markdown
## 下次检查清单

- [ ] 是否存在两个文件同时定义同一规则？
- [ ] 当前动作引用的是 active 规则，还是一次复盘里的中间判断？
- [ ] 写入前是否已确认用户授权和目标路径？
```

## 过度保留反例

坏例特征：

- 粘贴完整聊天、完整命令输出、完整工具日志。
- 每个尝试都保留同等权重。
- 用户要在噪音里自己找结论。

为什么坏：

- 信息密度低。
- 阅读成本高。
- 高价值判断没有被编辑出来。

改写策略：

- 原始证据只保留能支撑判断的片段。
- 其余写成“排除过的路径”或“无效尝试摘要”。

## 过度提炼反例

坏例特征：

```markdown
系统设计要保持单一事实源，并建立良好的反馈闭环。
```

为什么坏：

- 抽象正确，但失去原始场景。
- 未来不知道这条原则适用于什么问题。

改写策略：

- 保留触发场景、反例和适用边界。
- 原则后必须跟一个具体“下次怎么用”。

## 格式炫技反例

坏例特征：

- callout、表格、标签、wikilink 很多。
- 但同一信息重复出现在摘要、时间线、索引和结论里。
- 读者仍然不知道主线。

为什么坏：

- 版式没有服务结构。
- 用户会被装饰打断。

改写策略：

- 每个版式组件只承担一个职责。
- 表格用于比较和状态，不用于堆长段落。
- callout 只用于摘要、风险、决策、元数据和边界。

## 视觉版式失控反例

坏例特征：

- CSS snippet 改了颜色、图标和表格，但没有验证 Obsidian 里是否真的显示。
- 图标使用本地 SVG 相对路径做 `mask`，文件存在但渲染为空白。
- Mermaid 为了横向阅读写成一条超长 `flowchart LR`，超过正文宽度或被缩到看不清。
- 把“callout 使用规则”“CSS 规则”写进用户母文档正文。
- 用户只要求轻量美化时，擅自改页宽、字号、行高、正文主线或章节结构。

为什么坏：

- 代码存在不等于读者能看到。
- 规则污染用户笔记，会让母文档变成 Agent 内部操作手册。
- 图和样式没有服务理解，反而增加阅读负担。

改写策略：

读取 `obsidian-visual-patterns.md`，按其中的执行模式、错误做法和验证流程重做。反例库只用于识别问题，不维护另一套视觉规则。

## 自行补信息反例

坏例特征：

- 在用户未确认的情况下新增字段、模块或关系链接。
- 把未讨论确认的模块当作默认输出。
- 从网页无法识别作者或日期时，为了完整性自行猜测。

为什么坏：

- raw 信息会混入未经确认的二次判断。
- 用户无法分清哪些来自来源，哪些来自 Agent 自己补全。

改写策略：

- 只保留用户确认过的字段和模块。
- 无法识别的网页元数据写“不适用”。
- raw 阶段不默认添加 `相关` 和 wikilink。

## Agent-only 反例

坏例特征：

- 全篇都是“下个 Agent 应该读取 X、执行 Y、检查 Z”。
- 用户看不到故事线、价值和判断依据。

为什么坏：

- 笔记只服务机器续作，不服务人的理解。
- 用户不会愿意持续阅读。

改写策略：

- 第一屏写给人看。
- Agent 操作建议放到“下次怎么用”或“恢复入口”。
- 保留判断依据，不只保留指令。

## 信息很多但入口很硬

坏例特征：

- 开头先解释编号体系、文档读法和内部规则。
- 时间线非常完整，但用户不知道最重要结论。
- 派生索引、当前判断、当前方案和客观结论互相重复。

为什么坏：

- 信息量大不等于可读。
- 用户第一屏没有获得阅读回报。
- Agent 可能把复盘中的中间判断误当作稳定规则。

改写策略：

- 第一屏给“结论 + 使用场景 + 风险边界”。
- 时间线只保留转折点。
- 派生索引合并成“当前有效结论”和“未闭环动作”。
- 把内部编号降级为证据引用，不作为阅读主线。
