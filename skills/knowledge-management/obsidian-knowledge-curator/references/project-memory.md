# 项目母文档维护

本文件用于维护 skill 或项目的母文档。母文档不是总结稿，而是项目工作台：帮助用户和未来 Agent 快速恢复背景、架构、迭代、决策和未闭环事项。

## 触发场景

用户出现以下意图时读取本文件：

- 回顾某个 skill 或项目的线程内容。
- 维护项目上下文、母文档、迭代时间轴、决策索引或未闭环事项。
- 要求把项目背景、目标、能力、边界、踩坑、解法和决策整理成长期入口。
- 要求把新迭代写回已有母文档。

## 路径安全

维护母文档前先确认目标路径。

- 母文档路径不存在：停止，报告路径不存在，询问用户是否创建或改路径。
- 母文档文件不存在：停止，报告文件不存在，询问用户是否创建新母文档。
- 文件已存在且用户要求替代：必须确认替代的冻结编号和目标文件。
- 不要因为“这是母文档”就自动创建、覆盖或移动文件。

## 必备结构

母文档默认使用序号化章节：

```markdown
# 项目名称 项目上下文

## 1. 项目元信息
## 2. 项目背景
## 3. 项目目标
## 4. 项目架构
## 5. 项目边界
## 6. 当前能力
## 7. 项目迭代时间轴
## 8. 已确认决策清单
## 9. 当前未闭环事项
## 10. 带冻结编号的项目 Workflow
## 11. 后续工作建议
```

可以按项目规模增删，但不要删掉项目元信息、项目边界、迭代时间轴、决策清单和未闭环事项。

## 项目元信息

项目元信息至少包含：

- Skill 名称或项目名称。
- 业务域或项目域。
- 当前源码版本。
- 源码路径。
- 母文档路径。
- 当前冻结编号。
- 目标读者。
- 维护状态。
- 触发边界。

触发边界要说明何时使用、何时不应使用。若触发边界与 `SKILL.md` 相关，保持同步；不确定时先引用当前 `SKILL.md`，不要自行扩写。

## 项目架构

项目架构、每层架构的职能和边界、为什么这样设计，必须绑定在同一个条目里。

推荐表格：

```markdown
| 架构层 | 职能与边界 | 为什么这样设计 | 对应产物 |
| --- | --- | --- | --- |
| 用户真实需求层 | ... | ... | ... |
```

可以使用 Mermaid 展示总览，但 Mermaid 不能替代表格。图用于快速理解，表格用于精确定义。Mermaid 布局、错误做法和验证流程以 `obsidian-visual-patterns.md` 为准。

## 项目边界

章节标题使用“项目边界”，不要写成泛泛的“核心边界”。

固定分为三类：

- 能做什么。
- 不能做什么。
- 必须先确认才能做什么。

边界要可执行，避免只写原则。

## 项目迭代时间轴

时间轴采用倒叙，最新迭代在前。

标题格式：

```text
状态符号 ITER-XXXX | YYYY-MM-DD HH:mm | 标题
```

状态符号：

- `✅` 已闭环。
- `⏳` 未闭环。
- `🔁` 持续中。
- `❓` 等待确认。

如果无法从线程或来源中获得真实时分，写：

```text
YYYY-MM-DD 时间不可得
```

不要编造精确时间。

每个迭代固定为完整事件单元：

| 维度 | 要求 |
| --- | --- |
| 发生了什么 | 客观描述本轮变化。 |
| 问题 / 坑 | 说明暴露的问题，不能只写任务。 |
| 影响 | 说明为什么这个问题值得修。 |
| 解法 | 说明采取或计划采取的处理方式。 |
| 关键决策 | 把本轮形成的规则写清楚。 |
| 闭环状态 | 已闭环、未闭环、持续中或等待确认。 |

## 决策与未闭环事项

项目迭代时间轴、决策清单和当前未闭环事项是同一事件的三种投影，不是三个独立模块。

- 时间轴是事实源：记录事件、问题、影响、解法、关键决策和派生事项。
- 决策清单是索引：只摘出已确认决策，必须绑定来源迭代。
- 当前未闭环事项是索引：只保留 `未闭环`、`持续中`、`等待确认` 事项，已闭环事项回到时间轴，不保留在当前未闭环列表。
- 修改任意一处时，必须同步检查另外两处是否需要更新。
- 状态符号按关联事项反推：存在 `未闭环` 事项用 `⏳`；只存在 `持续中` 事项用 `🔁`；等待用户确认用 `❓`；没有开放事项才用 `✅`。

决策清单和未闭环事项必须绑定来源迭代编号。不要让用户在时间轴、决策表和待办表之间重新连线。

决策表：

```markdown
| 决策编号 | 来源迭代 | 决策 | 状态 | 影响范围 |
| --- | --- | --- | --- | --- |
| DEC-0001 | ITER-0001 | ... | 已确认 | ... |
```

未闭环事项表：

```markdown
| 事项编号 | 来源迭代 | 事项 | 状态 | 下一步 |
| --- | --- | --- | --- | --- |
| TODO-0001 | ITER-0008 | ... | 未闭环 | ... |
```

## Skill 修改后的母文档同步检查

如果任务修改了某个 skill 的源码、规则、reference、模板、样本、测试、版本或安装状态，必须同步检查该 skill 的项目母文档是否需要维护。

检查范围：

- 项目元信息：当前源码版本、源码路径、安装状态、维护状态。
- 当前能力：是否新增、确认、废弃或闭环了能力。
- 项目迭代时间轴：是否需要追加新 ITER，或更新来源 ITER 的状态符号和闭环状态。
- 已确认决策清单：是否新增、修正或废弃 DEC。
- 当前未闭环事项：是否新增、关闭、改状态或改下一步。
- 后续工作建议：是否仍然最新，是否包含已经完成的旧事项。

如果母文档路径存在且可写，按用户授权范围同步维护；如果路径不存在、文件不存在或本次没有写入授权，必须报告状态并说明需要用户确认，不能假装已经更新。

## 冻结编号 Workflow

母文档必须包含带冻结编号的项目 workflow。优先使用 Mermaid + 编号步骤表，保证 Mermaid 不渲染时仍可读。

Mermaid 只画主路径和关键反馈闭环；布局使用 `obsidian-visual-patterns.md` 的阶段式结构。编号步骤表要说明动作和输出。

## 预览格式

如果在对话中展示完整 Markdown 预览，外层代码块必须使用四反引号或更多，避免内部三反引号代码块提前结束预览。

```text
````markdown
内部可以安全包含 ```mermaid
````
```

也可以不使用外层代码块，改用“预览开始 / 预览结束”分隔线。
