# 信息类型与组织结构

先判断材料类型，再决定笔记结构。不要把所有内容都写成同一种总结。

## 类型路由

| 类型 | 适合沉淀的信号 | 默认结构 |
| --- | --- | --- |
| 踩坑 | 报错、误判、失败路径、重复排查、环境差异。 | 触发场景 -> 症状 -> 根因 -> 解法 -> 适用边界 -> 下次检查清单。 |
| 决策 | 取舍、否定方案、架构边界、长期偏好。 | 背景 -> 决策 -> 为什么 -> 被否掉的方案 -> 影响范围 -> 复核点。 |
| 复盘 | 一次事件、项目、讨论或事故的过程与结果。 | 30 秒读法 -> 时间线 -> 转折点 -> 根因/洞察 -> 防复发动作 -> 未闭环。 |
| 知识点 | 概念、事实、对象、稳定结论、方法说明。 | 定义 -> 为什么重要 -> 使用场景 -> 例子 -> 易错点 -> 来源或延伸材料。 |
| 想法 | 未验证洞察、产品灵感、研究方向、假设。 | 想法一句话 -> 触发来源 -> 价值假设 -> 反证风险 -> 下一步验证。 |
| 讨论 | 多方观点、分歧、共识、问题清单。 | 讨论主题 -> 共识 -> 分歧 -> 证据 -> 决定/待决 -> 后续动作。 |
| 偏好 | 用户长期表达风格、协作禁忌、工作习惯。 | 用户原话 -> 适用场景 -> Agent 行为要求 -> 边界 -> 反例。 |
| 交接 | 未完成任务、跨线程续作、当前状态。 | 目标 -> 当前状态 -> 已完成 -> 未完成 -> 阻塞 -> 下一步 -> 恢复入口。 |

## 拆分规则

一篇笔记只承担一个主职责。

当材料同时包含多种价值时，优先拆分：

- 事故过程和防复发 SOP 分开。
- 决策记录和执行清单分开。
- 用户偏好和单次任务复盘分开。
- 未验证想法和已确认知识点分开。

如果暂时不拆，正文中必须标明“本篇主线”和“派生待拆主题”。

## 标题规则

标题要传达可复用价值，不要只写日期和任务名。

| 弱标题 | 更好的标题 |
| --- | --- |
| `2026-06-22 项目复盘` | `不要把入口规则和同步规则混成同一个事实源` |
| `Agent 回写总结` | `Agent 回写前必须先判断：这是交接、决策还是经验` |
| `一次报错记录` | `Windows 路径解析失败时先检查 drive 是否存在` |

## 第一屏规则

第一屏不要先解释文档怎么读。先给用户阅读回报。

推荐第一屏：

```markdown
> [!summary] 30 秒读法
> - 这篇记录解决什么问题：
> - 最重要结论：
> - 下次遇到类似情况先做：
> - 当前可信度：
```

复杂时间线可以放在第二屏之后。先让用户知道为什么值得继续读。
