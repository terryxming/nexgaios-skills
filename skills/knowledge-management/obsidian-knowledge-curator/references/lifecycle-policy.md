# 版本与生命周期

本文件规定两类版本：skill 自身版本，以及每篇输出内容的冻结版本。

## Skill 自身版本

修改 `obsidian-knowledge-curator` 的行为规则、输出契约、评测标准或安全边界时，必须升级 `skill.yaml` 的 `version`，并更新 `CHANGELOG.md`。

修改源码、规则、reference、模板、样本、测试、版本或安装状态后，还必须读取 `project-memory.md`，执行“Skill 修改后的母文档同步检查”。检查对象至少包括项目元信息、当前能力、项目迭代时间轴、已确认决策清单、当前未闭环事项和后续工作建议。

修改上述文件时，还必须运行或满足 `impact.yaml` 定义的影响链路检查。关联 witness 文件没有同步修改时，必须提供 `docs/impact-reviews/obsidian-knowledge-curator/` 下的机器可读审查回执，说明已经检查且无需修改。

版本建议：

| 变更 | 版本动作 |
| --- | --- |
| 修错字、补说明，不改变行为 | patch |
| 新增输出规则、确认闸门、评测项、生命周期规则 | minor |
| 改变默认写入策略、字段契约或触发边界 | major |

仅补齐测试、样本、README 或 reference 路由，但会改变未来 Agent 的执行方式时，也要按行为修正处理，至少升级 patch 版本并记录 CHANGELOG。

已被用户否定或替代的旧规则必须删除或改写，不要在正文中并列保留两个互相冲突的规则。

## 输出内容冻结版本

冻结编号格式：

```text
OKC-YYYYMMDD-[开放式生成的主题]-NN
```

示例：

```text
OKC-20260622-[AWS MEMORY]-01
OKC-20260622-[AWS MEMORY]-02
```

编号规则：

- `YYYYMMDD` 使用笔记日期。
- `[主题]` 由当前内容开放式生成，使用 2-4 个英文或中文关键词。
- `NN` 从 `01` 开始，两位数递增。
- 用户反馈后生成的新版本必须递增编号。

## 内容生命周期

| 状态 | 含义 | 允许动作 |
| --- | --- | --- |
| 草案 | Agent 正在组织信息，尚未给用户完整预览。 | 可自由修改，不可写入。 |
| 冻结预览 | 已输出完整 Markdown 和冻结编号。 | 等待用户确认或反馈。 |
| 用户反馈中 | 用户提出修改意见。 | 先判断合理性并挑刺，再生成新冻结版本。 |
| 已确认 | 用户明确确认某个冻结编号。 | 可以按该版本写入。 |
| 已落盘 | 该冻结版本已经写入目标路径。 | 读回验证，不再静默修改。 |
| 已替代 | 后续冻结版本被确认并替代旧版本。 | 可覆盖同一文件或另存，取决于用户确认。 |
| 已废弃 | 用户否定该版本。 | 不写入，不再引用为当前版本。 |

## 用户反馈处理

用户提出反馈时，不要立即照单全收。

必须先输出：

1. 哪些反馈合理。
2. 哪些反馈可能有副作用。
3. 哪些反馈需要继续确认。
4. 新版本会删除什么、保留什么、替代什么。

然后生成新的冻结编号版本。

## 写入替代规则

如果用户确认“用新版本替代旧版本”，允许覆盖同一文件。

覆盖前必须确认：

- 目标文件存在。
- 用户确认替代的是哪个冻结编号。
- 新版本冻结编号已写入正文或 frontmatter。
- 过期模块和旧字段已删除，而不是并列保留。
