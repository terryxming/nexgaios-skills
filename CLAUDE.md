## 通用工程纪律

这个文件之所以存在，是因为大语言模型在写代码时会犯一些可预测的错误。不是随机错误，而是一遍又一遍出现的同一类错误。我看过太多次，所以把它们写了下来。

这些不是建议，而是规则。遵守它们，你就能写出不需要被重写的代码。忽略它们，你就会写出看起来很厉害、但在生产环境里会出问题的代码。

### 1. 写之前先读

大语言模型写出糟糕代码的最大原因，就是在写新代码之前没有阅读现有代码库。你看到一个任务，就把它匹配到训练数据里的某种模式，然后开始生成。这几乎总是错的。

在写任何东西之前：

- 阅读你即将修改的文件。不是扫一眼，而是认真读。
- 看看项目里类似的东西是怎么做的。如果 API 路由有既定模式，就遵循那个模式。如果已经有一个工具函数完成了你需要的一半功能，就使用它。
- 检查文件顶部的 imports。它们会告诉你这个项目实际使用了哪些库。如果项目到处都用 `fetch`，就不要引入 `axios`。如果项目使用原生方法，就不要引入 `lodash`。
- 查看测试文件。测试会告诉你真正期望的行为是什么，而不是你以为它应该是什么。

这里的失败模式很明显：你生成了“正确”的代码，但它和所在代码库完全格格不入。它能运行，但看起来像另一个人写的（因为确实是另一个实体写的）。于是人类要么重写它以匹配项目风格，要么永远忍受不一致。两种都不好。

如果你不确定这个项目里某件事是怎么做的，就说出来。“我在代码库里没有看到 X 的模式，我应该沿用 Y 的做法，还是采用别的方式？”这永远比猜测更好。

### 2. 编码之前先思考

不要在还没弄清楚自己到底要做什么之前就开始写代码。这听起来显而易见，但它是最常见的失败模式。

实践中应该这样做：

**说明你的假设。** 如果用户说“添加认证”，这可能意味着 session cookie、JWT、OAuth、basic auth，或者另外五种东西。不要悄悄替用户选一种。你应该说：“我假设你想要基于 JWT 的认证，带 refresh token，并存储在 httpOnly cookie 中。如果你想要别的方式，请告诉我。”如果你猜错了，只损失 10 秒。如果你沉默地猜错了，可能会损失一个小时。

**说清取舍。** 几乎每个实现选择都有取舍。如果你要加缓存，就说：“这会用内存换速度，同时引入缓存失效这个需要考虑的问题。”用户可能会说：“其实我不想要这种复杂度。”最好在你写 200 行代码之前就知道这一点。

**如果有多种方案，简要列出来。** 不要列五种。两种，最多三种，并给出推荐。“这有两种做法。方案 A 更简单，但不处理边界情况 X。方案 B 能处理所有情况，但会增加对 Z 的依赖。除非你预计 X 真的会发生，否则我会选 A。”

**如果有困惑，就停下来。** 不要用听起来合理的代码填补困惑。当你不理解需求时生成代码，结果通常是通过粗略 review 但在关键时刻失败的代码。直接说明哪里困惑，然后提问。

### 3. 简单性

写出能解决问题的最少代码。不是你能想象出来的、理论上能解决问题的最少代码，而是此时此刻真正解决这个具体问题所需要的最少代码。

过度设计的冲动很强。要抵抗它。下面是实践中过度设计的样子：

**过早抽象。** 你需要发送一种邮件。于是你写了一个 `EmailService` 类，用策略模式支持多个 provider、模板引擎和重试策略。用户想要的是 `sendWelcomeEmail(user)`。那就写这个函数。如果他们之后需要更多东西，他们会说。

```python
# 坏例子：你写了这个
class EmailService:
    def __init__(self, provider: EmailProvider, template_engine: TemplateEngine):
        self.provider = provider
        self.template_engine = template_engine

    async def send(self, template: str, context: dict, recipient: str, **kwargs):
        rendered = self.template_engine.render(template, context)
        await self.provider.send(recipient, rendered, **kwargs)

# 好例子：你本该写这个
async def send_welcome_email(user):
    body = f"欢迎 {user.name}！你的账号已经准备好了。"
    await send_email(to=user.email, subject="欢迎", body=body)
```

**猜测式错误处理。** 你给所有东西都包上 try/catch，处理那些不可能发生的错误。你验证来自自己代码、且已经在上游验证过的输入。你给永远不会是 null 的值加 null 检查。每一行错误处理，都是别人必须阅读和理解的一行代码。只处理那些真的可能发生的错误。

**不必要的可配置性。** 你把批处理大小做成参数。你把重试次数做成可配置。你为永远不会变化的东西添加环境变量。配置不是免费的。每一个配置项，都是别人必须做出的一个决策，也是别人必须正确设置的一个值。在没有真正理由之前，先硬编码。

**无用的灵活性。** 只有一个实现的接口。只有一个子类的抽象基类。只会被一种类型实例化的泛型参数。这些东西有成本：认知负担、间接层、更多需要跳转的文件。而在第二个实现真正出现之前，它们没有任何收益。

检验简单性的方法：把你的代码给一个不熟悉项目的人看。如果他们必须问“为什么这里要这样抽象？”，而答案是“以防以后需要……”，那你就过度设计了。“以防以后需要”不是需求。它只是对未来的猜测，而对未来的猜测通常是错的。

### 4. 外科手术式修改

当你编辑现有代码时，diff 应该尽可能小。你改动的每一行，都可能引入 bug，都需要别人 review，也会永远出现在 git blame 里。

规则：

**不要碰没有被要求碰的东西。** 如果你在修复函数 A 的 bug，同时注意到函数 B 的变量名很奇怪，别管它。如果函数 C 的注释有错别字，别管它。如果 import 顺序不符合你的偏好，别管它。你的工作是修复函数 A 的 bug。

**匹配现有风格。** 如果文件使用单引号，就使用单引号。如果文件使用 `snake_case`，就使用 `snake_case`。如果文件没有分号，就不要添加分号。如果文件使用 `var`（是的，即使在 2025 年），你的新增代码也使用 `var`，除非用户要求你现代化。文件内部的一致性比你的个人偏好更重要。

**清理你自己造成的问题，而不是替别人清理。** 如果你的改动导致某个 import 不再使用，就删除那个 import。如果你的改动导致某个变量不再使用，就删除那个变量。如果你的改动导致某个函数不再使用，就删除那个函数。但只有当这是你的改动造成的时候才这样做。已有的死代码不是你的问题，除非有人要求你清理。

**不要重新格式化。** 不要对一个原本没有用 prettier 格式化的文件运行 prettier。不要把缩进从 4 个空格改成 2 个空格。不要在 imports 原本没有按字母排序时把它们重新排序。重新格式化会制造巨大的 diff，掩盖你的真实改动，并让 code review 变得痛苦。

检验方法：看你的 diff。你能否说明每一行改动都和用户要求有直接关系？如果某一行存在的理由是“我刚好在这里，就顺手……”，那就还原它。

### 5. 验证

能工作的代码，和你以为能工作的代码，中间隔着测试。你应该对这个区别保持警惕。

**修 bug 时先写测试。** 在修复任何东西之前，先写一个能复现 bug 的测试。运行它。看它失败。然后修复 bug。再运行测试。看它通过。这不是可选项，也不是 TDD 教条。它是证明你真的修复了问题、而不是只让症状消失的唯一方式。

**在改动前后都运行现有测试。** 如果测试在你的改动前通过、改动后失败，那就是你弄坏了什么。这很明显。不那么明显的是：如果测试在你的改动前就已经失败，要说出来。不要默默忽略已有失败，然后让你的改动背锅。

**不要为了写测试而写测试。** 检查构造函数是否设置了属性的测试毫无价值。检查你的验证逻辑是否真的拒绝了坏输入的测试才有价值。测试行为，而不是实现。测试有意思的情况，而不是平凡的情况。

**如果你没法写测试，说明原因。** 有时架构让测试很难写。这是有价值的信息。“我没法轻松测试这个，因为数据库调用和业务逻辑紧密耦合”是一个信号，说明某些东西可能需要重构。不要只是跳过测试然后希望一切顺利。

### 6. 目标驱动执行

每个任务在你开始写代码之前，都应该有清晰的成功标准。如果标准模糊，就把它具体化。如果你无法具体化，就提问。

把模糊任务转化为可验证的任务：

- “添加验证”变成“拒绝 email 缺失或无效的输入，返回 400，并返回说明错误原因的消息，同时为两种情况添加测试”
- “修复 bug”变成“写一个复现报告行为的测试，让测试通过，并验证现有测试仍然通过”
- “提升性能”变成“先 profile，找到瓶颈，修复那个具体问题，然后再次测量”

对于任何超过一步的事情，在执行前先说明计划：

```
Plan:
1. Add the new database column with a migration
2. Update the model to include the new field
3. Modify the API endpoint to accept and return the field
4. Add validation for the field
5. Write tests for the new behavior
6. Run full test suite to check for regressions
```

这有两个作用：它让用户能在你浪费时间实现之前发现你的方案问题；同时它也强迫你真正想清楚步骤，而不是一头扎进去边做边想。

### 7. 调试

当某个东西不能工作时，不要猜。要调查。

**阅读错误信息。** 读完整条，包括 stack trace。大语言模型有一个很糟糕的习惯：看到一个错误，就根据错误类型立刻生成一个“修复”，而不是阅读错误到底在说什么。一个 `TypeError` 可能意味着一百种不同的事情。具体消息和 stack trace 会告诉你是哪一种。

**先复现。** 在你修改任何东西之前，确保你能复现问题。如果你不能复现它，就无法验证你的修复。“我觉得这应该能修好”不是调试，而是在赌博。

**一次只改一件事。** 如果你改了三件事，bug 消失了，你不知道是哪一件修好了它。你也不知道另外两件有没有引入新的 bug。改一件。测试。再改下一件。再测试。

**不要在不理解根因的情况下添加 workaround。** 如果某个值意外地是 null，不要只是加一个 null 检查然后继续。弄清楚它为什么是 null。null 检查也许能防止崩溃，但底层 bug 仍然存在，并会在之后以别的形式出现。

**如果你卡住了，就说出来。** “我试过 X 和 Y，都没有成功。这是我看到的现象。我认为问题可能是 Z，但还不确定。”这比默默随机尝试 20 轮有用得多。

### 8. 依赖

不要不加思考地添加依赖。

你添加的每个依赖，都是你无法控制的代码，并且会成为项目的永久组成部分。它需要维护、更新、做安全审计，也需要团队里的每个人理解。这个成本几乎总是比看起来更高。

在添加 package 之前：

- 你能用项目里已有的东西完成吗？如果项目已经有 `axios`，就不要添加 `node-fetch`。如果项目使用 `date-fns`，就不要添加 `moment`。
- 你能用标准库完成吗？你不需要为了 `Array.prototype.map` 使用 `lodash`。如果有 `crypto.randomUUID()`，你就不需要 `uuid`。
- 这个依赖真的还在维护吗？检查最后一次 commit 的时间。检查 issue 数量。检查维护者是否回复 issue。
- 它有多大？如果你为了格式化一个日期添加一个 500KB 的包，那通常不值得。

当你确实要添加依赖时，说明原因。“我添加 zod，是因为这个项目需要运行时 schema 验证，而现有依赖里没有能做这件事的工具”是可以的。默默往 `package.json` 里加包是不可以的。

### 9. 沟通

你如何沟通代码，和代码本身一样重要。

**说明你做了什么，以及为什么。** 不要只是丢一个代码块。“我把验证逻辑移到了单独的函数里，因为它在三个 endpoint 里重复出现。这样也能单独测试它。”这样用户不用读每一行代码，也能理解改动。

**标出担忧。** 如果你实现了用户要求的东西，但你认为这个方案有问题，要说出来。“这个能工作，但它会对列表里的每个 item 都做一次数据库调用。如果列表变大，会很慢。要我把它改成批量查询吗？”这种主动沟通可以在之后节省很多时间。

**精准说明你的不确定性。** “我不确定这个库是否支持 streaming responses”是有用的。“我觉得这应该能工作”没有用。区别在于，前者明确告诉用户需要验证什么。

**不要解释用户已经知道的东西。** 如果用户让你添加一个 REST endpoint，就不要解释什么是 REST。如果用户要求数据库索引，就不要解释索引是什么。让你的解释层级匹配用户已经展现出的知识水平。

**commit message 很重要。** 如果你在写 commit message，就写得具体一点。“Fix bug”没有用。“Fix null pointer in user lookup when email contains uppercase chars”能告诉下一个人到底发生了什么。

### 10. 常见失败模式

这些是我最常看到的模式。如果你发现自己正在做其中任何一种，就停下来重新考虑。

**厨房水槽。** 用户要求添加一个功能，你却“顺便”重构了半个代码库。不要这样。做那一件事。

**错误的抽象。** 你为一个只在一个地方存在的问题构建了一个漂亮的通用方案。重复远比错误的抽象便宜。在抽象之前，先复制粘贴两次。

**隐形决策。** 你做出了一个架构选择（数据库 schema、API 形状、认证策略），但没有把它标记为一个决策。这些选择很难逆转，用户应该知道你做了它们。

**乐观路径。** 你写的代码完美处理 happy path，却忽略或崩溃于其他所有情况。想想 API 返回 500 时会怎样。文件不存在时会怎样。用户提交空表单时会怎样。

**知识幻觉。** 你自信地使用一个不存在的 API、一个两版之前已经被移除的参数，或者一个你想象出来的库功能。如果你不是 100% 确定某个方法以这个精确签名存在，就说出来。查文档。查看项目中的实际源码。

**风格漂移。** 你用自己“偏好”的风格写代码，而不是匹配项目。OOP 代码库里写函数式模式。函数式代码库里写类。JavaScript 项目里写 TypeScript 风格。匹配代码库，而不是你的偏好。

**失控重构。** 你开始修一个东西。它牵扯到另一个东西。另一个又牵扯到下一个。二十分钟后，你改了 15 个文件，却不确定自己最初到底要做什么。如果一个修复开始连锁扩散，就停下来。告诉用户发生了什么。在继续之前先获得认可。

---

当这些准则能带来更少的无必要 diff、更少因过度复杂而导致的重写，并且澄清问题发生在实现之前而不是犯错之后时，它们就是有效的。
